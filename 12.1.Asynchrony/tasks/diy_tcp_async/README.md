# Event Loop and async TCP-server

В этом задании вы реализуете упрощённый аналог библиотеки `asyncio` и примените его для своего же tcp-сервера:

1. Event loop (класс `Scheduler`) с таймерами и ожиданием готовности файловых дескрипторов через `select`.
2. Примитивы для кооперативной многозадачности (`Task`, `await switch()`).
3. Асинхронный TCP-сервер, который:
   - принимает подключения,
   - читает нуль-терминированный байт-сообщение,
   - разворачивает его,
   - отправляет клиенту.

## Структура задания

Что имеем и что придется сделать:

- `scheduler.py` — ваш event loop:
  - класс `Scheduler`;
  - класс `Task`;
  - класс `Awaitable` и функция `switch`;
  - глобальный объект `sched = Scheduler()`.
- `server.py` — асинхронный TCP-сервер:
  - `async def tcp_server_async(addr)`;
  - `async def echo_handler(sock)`;


### Важно!

Класс `Task` — оболочка над корутиной:

1. хранит `self.coro` — саму корутину;
2. по вызову `task()`:\
  2.1. присваивает `sched.current = self`;\
  2.2. делает один шаг корутины: `self.coro.send(None)`;\
  2.3. если корутина не завершилась (`StopIteration` не возник) и `sched.current` не был сброшен:
   - 2.3.1. кладёт себя обратно в `sched.ready`, чтобы продолжить позже;

   2.4. если корутина завершилась, таск больше не попадает в очередь.

Асинхронные `sleep/recv/send/accept` должны:

1. зарегистрировать текущую задачу в соответствующих структурах (`call_later`, `read_wait`, `write_wait`);
2. поставить `sched.current = None`;
3. сделать `await switch()` — то есть, отдать управление планировщику.

`switch()` возвращает примитивный `Awaitable`, чей `__await__` просто `yield`-ит один раз. Это ровно тот момент, когда управление возвращается в `Scheduler.run()` и он может переключиться на другие задачи.

Обратите внимание, что TCP — потоковый протокол:

1. сообщение может прийти кусками (часть в одном `recv`, часть в другом);
2. нельзя надеяться, что первый `recv` не будет пустым;
3. нельзя надеяться, что один `recv` вернёт всё сразу, даже для маленьких сообщений;
